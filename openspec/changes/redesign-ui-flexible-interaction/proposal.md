## 为什么

当前界面能力以功能可用为主，但在整体交互上存在三类限制：

- 布局结构固定，难以适配写作、校对、发布等不同任务情境。
- 颜色与视觉缺乏系统化约束，主题切换与界面一致性难以长期维护。
- 可用性评估缺少明确任务导向，用户完成核心流程时存在认知负担。

该变更旨在将产品从“功能集合”升级为“任务导向的交互系统”，在不牺牲稳定性的前提下提升灵活性与效率。

## 变更内容

- 重构整体信息架构与界面分区，建立统一的工作台布局模型（导航区 / 主工作区 / 上下文区 / 状态栏）。
- 引入渐进式布局灵活性：默认预设布局 + 可折叠 + 可记忆（后续支持多 Workspace）。
- 建立语义化主题令牌体系（颜色、间距、边框、层级），统一浅色/深色/跟随系统行为。
- 以核心任务链路定义可用性标准：导入/写作 → 样式调整 → 预览校验 → 导出/发布。
- 明确交互反馈规范：主动作层级、处理中状态、错误恢复、可发布性提示。

## 功能 (Capabilities)

### 新增功能

- `adaptive-workbench-layout`: 提供可配置但有边界的工作台布局能力，支持不同任务场景切换。
- `semantic-theme-tokens`: 基于语义令牌的主题系统，保证颜色与组件行为一致。
- `task-oriented-usability`: 以任务完成效率为目标的可用性交互规范与验收标准。

### 修改功能

- `note-preview-interaction`: 将预览交互并入新的工作台模型，统一布局与状态反馈语义。

## 范围与边界

- 本变更聚焦 UI/UX 架构与交互规范，不实现业务能力新增。
- 允许对现有页面结构和视觉规范进行重排，但不改变核心发布能力的业务语义。
- 旧样式与新样式可在过渡期并存，但必须有明确迁移计划与回退策略。

## 影响

- 受影响模块（预估）：
  - `packages/frontend/src/components/**`（主界面分区、面板、工具栏、状态栏）
  - `packages/frontend/src/index.css`（主题令牌、布局基线、组件层级样式）
  - `packages/frontend/src/store/**`（布局状态、主题状态、用户偏好持久化）
  - `packages/obsidian/**`（宿主容器适配、主题同步边界）

- 受影响行为：
  - 页面信息架构与操作路径
  - 主题切换与视觉一致性
  - 核心任务流的完成效率与错误恢复体验
